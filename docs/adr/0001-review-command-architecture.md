# ADR-0001: 合同审阅采用新命令而非 Agent 通用流程

## 状态
已采纳

## 背景

V3 已有一个 `/agent <任务>` 通用智能体命令（基于 LangGraph 规划-执行-反思循环）。对于「审合同」这个具体场景，有两种实现方式：

1. **方案 A（Agent 模式）**：用户用 `/agent 帮我审这个合同`，让 Agent 自主规划审阅步骤
2. **方案 B（专用命令）**：新增 `/review <合同.pdf>` 命令，走确定性流程

## 决策

采用 **方案 B（专用命令 `/review`）**。

## 理由

| 维度 | 方案 A (Agent) | 方案 B (/review) |
|------|---------------|-----------------|
| **流程确定性** | Agent 每次需要推理"先做什么"，可能每次步骤不一样 | 固定流程：加载→提取→分析→报告，可预测 |
| **执行速度** | 每次多步推理和 LLM 调用（规划+执行+反思） | 确定性管道，更少 LLM 调用 |
| **输出一致性** | Agent may format output different each time | 严格流程控制，输出格式一致 |
| **审计专业要求** | 审计底稿要求可追溯、可复核，Agent "自由发挥"不满足 | 确定性流程保证每一步可追溯 |

## 影响

- 正面：执行更快、输出一致、审计可追溯
- 正面：与现有 `/agent` 关注点分离（Agent 做开放任务，/review 做审计流程）
- 负面：需要额外开发 `/review` 命令及其管道，无法直接复用 Agent 的 LangGraph 编排
- 负面：未来扩展到其他审计程序（分析性复核、控制测试）需要类似的新命令

## 备选方案

考虑过 **混合方案**：用 Agent 做"任务分解"，把审合同拆成子任务，但确定性部分（结构化提取）仍然走管道。放弃原因：增加了不必要的复杂度，审计场景不需要开放式的任务分解。

## 后续

- 本 ADR 不影响现有 `/agent` 命令的继续使用
- 后续如增加「分析性复核」「控制测试」等审计程序，按相同模式新增专用命令
