# 酒店负责人个人经营治理工作台使用说明

## 启动

```powershell
python -m pip install -r requirements.txt
streamlit run enterprise_app.py
```

原入口仍兼容：

```powershell
streamlit run audit_app.py
```

## 每日经营驾驶舱

操作台默认进入“每日经营驾驶舱”。负责人可以维护 PMS 日报来源表头并上传 CSV/XLSX；系统在本机完成原件归档、营业日版本化和确定性指标计算。

首版展示可售房、已售房、入住率、客房收入、平均房价（ADR）和每间可售房收入（RevPAR）。多行日报必须映射唯一“客房库存分段”，合计、小计、汇总、重复分段及异常数字会在计算前阻断。每项结果均保留来源文件、日报编号、日报版本和指标口径版本。

## 已实现流程

所有流程默认对照本板块知识资料、公司级章程与历史案件；结果保存命中文档、版本、原文分块、相关度和对照意见。

| 领域 | 流程 | 主要结果 |
|---|---|---|
| 审计与合同 | 商业合同审阅 | 合同要素、条款缺失、异常比例、底稿 |
| 人力 | 劳动合同审阅 | 期限、工资、工时、社保、试用期、竞业限制 |
| 人力 | 招聘匹配 | 技能/经验/项目/行业证据评分，盲审，不自动淘汰 |
| 行政 | 制度审阅 | 版本、范围、归口、审批、职责分离、例外、留存 |
| 行政 | 会议事项 | 决定、负责人、截止日期、逾期提示、任务同步 |
| 财务 | 费用审阅 | 重复票据、限额、预算、附件、日期、拆分、审批冲突 |
| 财务 | 预算分析 | 执行率、余额、差异、超支、零预算支出、低执行 |

任务中心同时保存会议行动项、人工任务和最终动作审批。审批强制申请人与审批操作人不同，并记录批准/驳回意见。

## 数据保存

默认数据目录为 `data/enterprise/`：

```text
data/enterprise/
├── enterprise.db       # 案件、执行、发现、任务、复核、审计日志
├── knowledge_sources/  # 制度、章程等知识资料原件
├── uploads/<case_id>/  # 每次上传的原始文件
├── reports/            # Markdown 报告
└── samples/            # 系统管理页生成的虚构标准样本
```

上传文件名会被清理，文件内容保存后计算 SHA-256。执行失败也会保留案件和错误信息。

## 本地模型

1. 安装并启动 Ollama。
2. 拉取模型，例如 `ollama pull qwen2.5:7b`。
3. 在 `.env` 配置：

```dotenv
OLLAMA_BASE_URL=http://localhost:11434/v1
OLLAMA_MODEL=qwen2.5:7b
```

在业务页面勾选“启用 Ollama/本地模型补充管理摘要”。不勾选时，文档解析、规则和财务计算仍然运行。

外部模型不会自动回退。只有本次操作明确授权后，系统才会先脱敏，再调用 `ENTERPRISE_EXTERNAL_PROVIDER` 指定的服务，并把模型路线记录为 `external-redacted`。

## 标准样本

在“系统管理”点击“生成或更新标准样本”，会生成：

- 20 份劳动合同；
- 5 份岗位说明和 30 份简历；
- 10 份制度；
- 20 份会议纪要；
- 200 条费用；
- 49 条预算与实际数据；
- `expected.json` 预期异常清单。

## 人工控制

在“历史与复核”中可查看附件、结果、证据和报告，逐条记录“已接受、已驳回、待补充材料、已整改”等人工结论。系统不会替代有权限人员做最终业务决定。

当前版本按单机 MVP 设计，"当前操作人"用于留痕和职责分离演示，不是正式身份认证。部署到多用户局域网前，应增加统一登录、服务端 RBAC、数据库备份和不可篡改审计日志。

## 知识资料与历史记忆

“知识资料库”可新增公司章程、部门制度、业务规范、会议决议和其他参考资料，并按审计、人力、行政、财务或公司级范围检索。案件经负责人标记为“已确认”或“已关闭”后，系统才把摘要、复核状态与主要发现按案件编号沉淀为“历史案例”；下一次同类分析会同时召回制度依据与已确认历史问题。

系统管理的酒店集团规划基线仅用于演示和需求规划，不代表任何企业的现行制度。规划详情见 `docs/hotel-management-rag-plan.md`。

## 系统配置与运行监控

系统管理页面可以直接维护 L1/L2/L3 的名称、资料范围、工作流等级映射和模型路由策略，并配置本地模型、外部 OpenAI 兼容服务、API Key、Base URL、模型名、超时及默认知识召回条数。普通配置保存在本地 SQLite 中并即时生效，优先级高于环境默认值；API Key 在 Windows 使用当前用户 DPAPI 加密，在其他系统写入系统 Keyring，数据库只保存引用。

运行监控页面分为当前任务、操作日志和模型调用。工作流执行、知识维护、任务、审批、复核、样本生成和配置变更会记录开始与结果；模型事件额外记录服务商、模型、调用路线、敏感等级、是否脱敏、耗时和错误，但不记录 API Key。
