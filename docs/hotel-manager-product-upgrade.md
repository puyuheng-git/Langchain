# 酒店负责人个人经营治理工作台升级评估

## 负责人结论

当前项目已经具备可靠的“材料归档—确定性分析—知识对照—人工复核—任务审批—历史留痕”底座，但产品主线仍是企业职能文件审阅。它可以作为技术 MVP，却还不是酒店负责人每天愿意打开的个人经营产品。

下一阶段不应继续横向增加通用 AI 功能，而应把产品重构为一条高频闭环：

```text
导入经营日报
  → 校验营业日、口径和数据质量
  → 计算全店经营指标与未来预测
  → 排出 3—5 条经营关注事项
  → 下钻证据、原因和管理工具
  → 形成个人行动卡
  → 用后续指标验证行动效果
```

首版的北极星标准是“十分钟经营闭环”，不是生成更长的报告，也不是增加模型调用。

## 已确认的产品边界

- 垂直服务中端或高端全服务商务酒店。
- 目标用户是单体酒店负责人、店总或总经理，首版不做集团协同。
- 本地优先、单用户、离线可用，用户主动执行加密备份或迁移。
- 经营数据通过 PMS、POS 和财务系统导出的 Excel/CSV 进入产品。
- 统一使用 PMS 夜审后的营业日；同一营业日重复导入形成可追溯版本。
- 首页覆盖客房、餐饮、宴会或会议、人工、能耗、费用和预算。
- 首页先给出经营关注事项和行动，再展示完整指标。
- 异常依据酒店自身预算、目标、历史同期、近期同星期、滚动预测和个人阈值确定。
- 未来预测以在手预订和历史预订增量曲线为核心，AI 只负责解释。
- 现有七个流程保留为管理工具箱，由关注事项触发并回流个人行动卡。

## 当前能力与主要缺口

| 能力 | 当前状态 | 负责人视角缺口 |
|---|---|---|
| 文件与案件 | 支持上传归档、哈希、解析、执行历史和报告 | 没有营业日、酒店、数据来源、导入批次和报表版本模型 |
| 专项分析 | 已有合同、劳动合同、招聘、制度、会议、费用、预算七个流程 | 流程彼此并列，未从经营异常进入，也不回到经营结果 |
| 财务计算 | 费用和预算使用确定性代码计算 | 缺少酒店收入、客房、餐饮、宴会、人工、能耗和利润指标 |
| 知识资料 | 支持部门资料、效力层级、版本和历史案件召回 | 缺少制度替代关系、冲突提示、到期复审和酒店专项知识模板 |
| 发现与复核 | 有高/中/低发现项、证据和人工状态 | 只偏向风险；缺少收入机会、影响金额、优先级和效果复核 |
| 任务与审批 | 有任务、截止日期和最小职责分离 | 缺少关注事项关联、预期目标、复盘指标和周期性提醒 |
| 模型路由 | 本地模型优先，外部调用需明确授权并脱敏 | 模型摘要尚未围绕店总决策语言、可解释原因和行动建议评测 |
| 运行与安全 | 有运行事件、模型事件和最小审计日志 | 本地数据库未形成加密、自动备份、恢复验证和迁移机制 |
| 产品界面 | Streamlit 已有首页、职能模块、知识库、任务、历史和系统管理 | 首页不是每日经营入口，导航仍以职能模块和系统能力为中心 |

## P0：先形成可每天使用的客房经营闭环

### 1. 重做首页和导航

首页固定为每日经营驾驶舱，默认展示：

- 数据新鲜度、最新营业日和导入质量；
- 昨日结果、今日重点、未来 7 天和 30 天预判；
- 3—5 条按影响和紧迫度排序的经营关注事项；
- 今日到期、逾期和待复盘的个人行动卡；
- 客房、餐饮、宴会、人工、能耗和费用的下钻入口。

合同、用工、制度等旧页面移动到“管理工具箱”，系统配置和运行日志移动到次级导航。

### 2. 建立酒店经营数据模型

当前 `cases`、`executions` 和 JSON 结果适合文件审阅，不适合跨营业日趋势计算。建议增加明确的经营实体：

- 酒店档案与客房库存；
- 营业日与经营日历；
- 导入批次、原始文件、字段映射和日报版本；
- 指标事实、预算目标、经营基线和口径版本；
- 在手预订快照、预订变更和预测版本；
- 经营关注事项、个人行动卡和效果复盘。

首版不需要用户、租户、组织和复杂权限表。单用户身份可以由本地个人档案表达。

### 3. 建立可复用的日报导入中心

- 提供 PMS、POS、财务报表的模板识别和字段映射向导。
- 第一次映射后保存为本地模板，后续拖入同类文件即可复用。
- 校验营业日、酒店名称、币种、房量、金额、空值和合计关系。
- 对字段缺失、跨报表日期冲突、负数异常和口径变化给出阻断或警告。
- 同一营业日重新导入时生成新版本，展示差异后由负责人确认生效版本。
- 每个指标可回到导入批次、文件、工作表、行列和原始值。

首版最重要的兼容性不是支持更多文件格式，而是让真实酒店的三至五种固定日报稳定重复导入。

### 4. 先实现客房权威指标引擎

首批指标包括：

- 可售房、已售房、入住率、免费房、自用房、维修房；
- 客房收入、ADR、RevPAR；
- 预订、取消、No-show、平均入住时长；
- 渠道、房型、客源结构和协议客户贡献；
- 在手预订、未来入住率、预订增量和收入预测；
- 相对预算、去年同期、近期同星期和滚动预测的偏差。

所有指标必须有公式、口径版本和来源链路。AI 不得覆盖计算结果。

### 5. 生成经营关注事项

关注事项不能只是“指标变红”，应包含：

- 发生了什么；
- 与哪条经营基线相比；
- 影响房量、收入、成本或利润的估计；
- 支撑判断的原始数据；
- 可核验的可能原因；
- 建议动作和建议完成时间；
- 负责人接受、忽略或调整的反馈。

排序建议使用确定性评分：预计影响 × 紧迫度 × 可干预性 × 置信度。AI 负责把结果转成简洁管理语言，不负责改变排序事实。

### 6. 升级任务为个人行动卡

在当前任务字段之外增加：

- 来源关注事项；
- 预期改善指标和目标值；
- 开始、截止和复盘日期；
- 行动前基线；
- 完成后的实际指标变化；
- 有效、无效、无法判断的复盘结论。

这样产品才能回答“采取了什么措施，以及措施是否改善经营”，形成个人管理记忆。

## P1：扩展为全店经营总览

### 餐饮经营

- 分餐厅、时段和品类的收入、客流、客单价与折扣；
- 食材成本率、毛利率、损耗、赠送和异常退单；
- 早餐住客覆盖率、包价拆分以及客房与餐饮联动；
- 与预算、历史同期、星期和宴会影响的对比。

### 宴会与会议

- 询价、暂定、确认、执行、结算和取消的销售漏斗；
- 场地利用率、预订收入、客单、毛利、定金和应收；
- 未来档期冲突、低利用日期和高价值机会提醒；
- 宴会合同审阅与经营预测直接关联。

### 人工与组织效率

- 编制、在岗、缺编、排班、出勤、加班和外包；
- 人工成本率、每间可售房人工成本、每间已售房工时；
- 业务量与排班错配、连续超时和关键岗位空缺；
- 招聘匹配降为辅助功能，人员效率和用工风险升为主功能。

### 能耗、费用与利润

- 水、电、气单耗和单位已售房能耗；
- 费用预算、异常增长、重复支出和预算外支出；
- 部门利润、GOP、GOPPAR 与现金压力；
- 收入增长但利润恶化等跨模块关注事项。

## P1：把现有功能升级成酒店管理工具箱

| 现有功能 | 建议升级 |
|---|---|
| 商业合同审阅 | 增加酒店采购、工程、租赁、OTA/渠道、宴会和外包合同模板；关联供应商、预算、付款节点和行动卡 |
| 劳动合同审阅 | 关联岗位、排班、证照、试用期、续签和用工成本；提供到期提醒 |
| 招聘匹配 | 降低导航权重，聚焦关键岗位胜任力、缺编影响和入职后效果复盘 |
| 制度审阅 | 增加制度替代/废止、冲突检测、适用酒店、责任人和复审到期提醒 |
| 会议事项 | 直接生成个人行动卡，识别重复延期，并关联经营关注事项和复盘指标 |
| 费用审阅 | 关联预算、合同、供应商、验收和营业日；增加跨期趋势与异常供应商分析 |
| 预算分析 | 从一次性表格分析升级为月度目标、每日分解、滚动预测和偏差归因 |
| 知识资料库 | 增加酒店 SOP、品牌标准、安全许可、食品卫生、工程维保和岗位标准模板 |

## P2：负责人仍会需要的增值功能

- 每日晨报：在应用打开时生成一页管理摘要，并允许导出 PDF、Markdown 或图片。
- 经营日历：标记节假日、展会、团队、婚宴、维修和本地重大活动，解释异常并辅助预测。
- 目标与情景模拟：模拟房价、入住率、渠道结构、人工和成本变化对收入与利润的影响。
- 客户声音：导入 OTA 点评和宾客反馈，聚类服务问题并关联部门行动。
- 供应商与合同台账：合同期限、付款节点、履约异常、价格变化和续约提醒。
- 证照与安全日历：消防、食品、特种设备、公共场所证照及检查整改提醒。
- 个人周报和月度复盘：总结关注事项、行动成效、重复问题和未兑现目标。
- 可选本地通知：营业日报未导入、重要异常、行动到期和证照到期提醒。

## P2：本地产品必须补齐的可靠性能力

- 数据库和原始文件的加密存储，密钥不与备份放在一起。
- 自动滚动备份、手工导出、恢复演练和版本升级前备份。
- 数据目录迁移、完整性检查和损坏恢复提示。
- 数据保留与清理策略，删除时明确影响范围并可在短期内恢复。
- 应用锁或操作系统凭据保护，避免电脑被借用时直接读取敏感信息。
- 模型和 API 密钥使用操作系统安全存储，不以普通数据库字段保存明文。
- 规则、指标、字段映射和预测模型均有版本，历史结果保持原口径可复算。

## 技术结构升级建议

### 深化领域模块

建议把当前通用 `ReviewWorkspace` 保留为管理工具箱协调器，新增独立的酒店经营模块：

```text
enterprise/hotel/
├── imports/       # 模板识别、字段映射、校验、版本
├── calendar/      # 营业日、节假日、活动
├── metrics/       # 客房、餐饮、宴会、人工、能耗、利润
├── baselines/     # 预算、同期、星期、阈值
├── forecasting/   # 在手预订与预订增量
├── attention/     # 关注事项生成、排序和反馈
└── actions/       # 行动卡与效果复盘
```

模块应对 UI、SQLite 和模型网关保持独立接口，使计算可被测试、命令行和未来界面复用。

### 拆分界面层

当前 `enterprise_app.py` 同时承担导航、表单、业务执行、历史、任务、配置和监控展示。新增驾驶舱后应按页面和可复用组件拆分，避免酒店指标继续堆入单文件。

### 引入数据库迁移

当前通过 `CREATE TABLE IF NOT EXISTS` 初始化表结构，适合 MVP，但无法可靠演进经营数据模型。新增经营表之前应建立可重复的 schema migration，并为升级和失败回滚编写测试；备份恢复测试随 P2 本地可靠性能力一并交付。

### 建立计算与预测评测集

现有自动化测试覆盖七个流程、安全、知识检索和监控，但需要增加：

- 营业日跨午夜与跨月边界；
- 同一日报重复导入和版本切换；
- 字段映射、缺失值、合计不平和口径变化；
- 入住率、ADR、RevPAR、GOPPAR 等权威公式；
- 同期、同星期、预算和阈值偏差；
- 在手预订、取消和预订增量预测；
- 关注事项排序、去重和影响估计；
- 行动前后指标效果复核；
- 备份、恢复、迁移和数据库损坏场景。

## 建议交付顺序

1. 营业日、导入批次、日报版本和字段映射。
2. 一种真实 PMS 日报的客房指标闭环。
3. 多基线比较与 3—5 条经营关注事项。
4. 个人行动卡和效果复盘。
5. 在手预订导入、7/30 天预订增量预测。
6. 餐饮、宴会、人工、能耗和利润模块。
7. 现有七个流程与关注事项、行动卡的关联升级。
8. 加密备份、恢复、迁移和可选本地提醒。

每一步都应形成可独立使用的纵向闭环，不应先一次性建设所有指标和所有报表模板。

## 首版验收指标

- 负责人可在 3 分钟内完成固定日报导入。
- 已保存模板的字段自动映射率达到 90% 以上。
- 日期或关键口径冲突不会静默进入指标计算。
- 所有权威指标可追溯到营业日、导入版本和原始单元格。
- 首页稳定给出不超过 5 条、可解释且可行动的重点。
- 负责人可在 10 分钟内完成复盘、确认重点并建立行动卡。
- 行动卡完成后能够判断关联指标是否改善。
- 目标用户每周主动使用至少 5 天。

## 当前不应优先建设

- 多租户、集团级组织权限和复杂协同审批；
- 大规模云端基础设施和强制账户体系；
- 同时对接多个 PMS/POS 厂商 API；
- 通用 Agent、自主执行代码和无边界的聊天入口；
- 以微调模型代替指标、规则和数据质量建设；
- 面向所有酒店类型的通用模板；
- 与每日经营闭环无关的更多横向文件审阅功能。

这些能力并非永远不做，而是在个人版证明高频经营价值之前不应消耗主要资源。
